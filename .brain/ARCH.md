# Architecture

## Current

- Next.js 16 App Router application.
- React 19 + TypeScript.
- Tailwind CSS 4 via PostCSS.

  an indeterminate progress bar, and rotating historical facts. Custom 404 at
  `src/app/not-found.tsx` reuses sleeping Coco with a small Space Mono message.
 Landing page at `/` (hero with Space — start cue → `/roll-call?view=new|returning`, Esc returns to the Space hint; meaning section with full-bleed `ParticleWorldMap` + etymology overlay; voyage section with `ParticleSolarSystem` and cycling planet verses; `PixelWaveHero` is a full-page continuous atmosphere behind all sections); `/roll-call` is a single tabbed auth screen (New player sign-up / Returning player profile picker from the `User` table, teachers first) on a `SparklingStars` black background with minimal chrome; new accounts are created through `/api/demo-auth/sign-up` (teachers get a default classroom with a join `code`, students join via that ship code); sign-up and returning-player sign-in both route teachers and students to `/nexus`; the demo session cookie `sapiens_demo_session` resolves against that table; shared top `AppNavbar` (`src/app/components/nav/`) sits in document flow (not overlay) on `/nexus`, `/ship`, `/sail/[slug]`, and `/home` with Nexus / Ship / Home links plus account pfp (sign-out menu); space theme (dark) on nexus/ship/home and light theme (white bg, dark text) on `/sail/[slug]`; slugless `/sail` demo is archived under `bin/`; lounge at `/ship` (`Lounge` + `HomeBackground` from ship scene, Coco from ship-2, space-themed `DialogueBox` on `canon/intro`; - Dialogue UI lives in `src/app/components/dialogue/`: `DialogueBox` is the lowest-level presentational text/choice/end view; `DialoguePanel` renders that view with shared header, portrait, and atmosphere UI; `StoryDialogue` owns story sessions and companion placement. The sail-only `FieldCompanion` (private multi-note journal in the notes pane: Save creates a draft `FieldNote` with content `{ kind: "private", html }`, New starts another draft, existing notes auto-save via PATCH, and source-backed COCO answers also create a read-only private draft (`authorType: "coco"`) for the current voyage with a collapsible sources list (same pattern as `/home` visitor notes; PATCH rejects content edits); private user notes can Share to Starstream as class `post`s (plain-text body, no voyage completion), and Coco notes Share as `visitorNote`s with optional commentary like `/home` actor notes; listed under the editor; no localStorage; Esc while COCO is generating cancels the request and resets, while Esc during answer paging leaves the overlay open), `VoyageTakeawayNote` (end-of-voyage field note under the report: Save → private Discoveries draft `{ body }`, Publish → class Starstream; required for assigned completion, optional for solo classroom cadets), `NotesEditor`, and Lexical `CalloutNode` live in the sibling `components/fieldnotes/` feature folder; the companion consumes the shared `DialoguePanel`. Themes: `vanilla` (white / Playfair+Manrope) and `space` (black / Space Mono). Nexus teacher and student views live in `src/app/nexus/`, driven by the demo session user and `/api/nexus` database snapshots.
- The learning MVP domain is persisted through `User`, `Classroom` (join `code`, Free/Restricted source policy), `ClassroomAssignment`, `Journey`/`JourneyVoyage`, `FieldNote` (private journal: user/bot author type and optional sources), `StarstreamLog` (public classroom/solo forum posts with `type` `post`|`visitorNote`, `allowReplies`, threaded `parentId` replies, URL/video `attachments`, and `StarstreamLike`s), and `VoyageProgress` in Prisma. `VoyageProgress` is keyed by student and story; `assignmentId` is optional so assigned and student-owned solo voyages share the same completion record. Assigned completion still requires a published class takeaway note (`{ body }`, not a private journal or visitor/Coco share), while solo completion is available to the voyage owner without an assignment. The progress write endpoint treats `storyId` as authoritative, resolving the viewer's current assignment for assigned voyages and falling back to owner-only solo access. Demo sign-in uses the `sapiens_demo_session` cookie; route guards resolve that cookie into a `User` row. Learning APIs live under `/api/classrooms`, `/api/assignments`, `/api/journeys`, `/api/field-notes`, `/api/progress`, and the Nexus adapter. Publishing a field note upserts a root `StarstreamLog`; unpublishing removes it. Nexus Starstream feeds load `starstreamLogs` (roots with nested replies and like state), including author-owned posts and classroom `visitorNote`s even when `assignmentId` is null (common for `/home` shares). Captains and cadets can like posts and reply one level deep on roots with `allowReplies` via Nexus actions `toggle-starstream-like` and `create-starstream-reply` (moderated, max length shared with the client). Captains and cadets can both share `/home` visitor notes as `visitorNote` posts; classroom Starstream includes visitor notes from the teacher and student members even when `assignmentId` is null. Student-owned voyages and notes use the separate Solo Voyages stream. Nexus also returns classroom `crew` (Captain + Cadets) and teacher-only `cadets` progress rows (assignment completion + latest Starstream post) for the Cadets and Crew workspace sections. Student Nexus snapshots include `completedAt` and collectible asset data on completed voyages for the Discoveries tab (field notes + earned collectibles, newest completion first). The Nexus teacher composer opens Specific voyage in Imagine mode by default; its free-text request is sent to Curator, whose returned brief pre-fills a reviewable manual draft. Publishing reuses that brief and routes directly to Director/story generation without rerunning Curator. Teacher-pasted exact source URLs bypass Researcher and become the only grounding set, while lesson-plan text is passed to Director as an explicit learning-focus constraint.
- Dialogue is data-driven JSON + a UI-independent `DialogueEngine` (`src/dialogue/`), rendered with React (not Phaser text, which blurred).
- Content lives under `src/content/`: `canon/` for shared framing dialogue (no voyage stats by default), `voyages/` for playable scenario graphs (opt into meters with `metadata.stats`), `art/` for atmosphere assets.
- Shared play snapshot lives in `src/game/state.ts` so React and a future Phaser world can read the same dialogue state.
- Orchestrator root keeps `run.ts`, `types.ts`, and `config.ts`; agent implementations live in `src/lib/orchestrator/agent/`; shared tool helpers (Wikipedia handlers, usage telemetry) live in `src/lib/orchestrator/tools/`. The generation pipeline runs researcher → director → writer → artist. Flourish source policy lives in `src/lib/orchestrator/agent/flourish.ts`: Free mode (empty approved-domain list) selects three primary sources including a best-fit Wikipedia article; any non-empty classroom domain list forces Restricted mode and validates against those hosts. Candidate URLs are preserved in progress. Writer receives only the approved source set and returns a factual/fiction report with sources and optional further reading, persisted on `Story.report`. The Writer selects character assets actually needed by the dialogue, while the Director's collectible is passed through to Artist. Model-backed agents share execution context, retry failures up to `ORCHESTRATOR_CONFIG.maxTries` total attempts, and receive the prior failure in retry instructions.
- Home reads the active classroom policy through `/api/classrooms/current` for either the classroom teacher or a student membership. Generation routes independently re-resolve that policy server-side so student requests cannot fall back to Free mode; Home preserves structured agent progress details for policy and candidate-source events.
- Curator is an optional story-entry stage. The Coco homepage sends `Something...` genre requests as free text through a shared dropdown/autocomplete option to Curator, which extracts any genre/bounds and selects one historical idea before immediately starting the story pipeline. `Take me to...` offers two predefined historical destination strings plus inline custom typing; these continue directly to the existing story generation pipeline. Curator may additionally return nullable, optional teacher-facing voyage metadata for Nexus and solo exploration; Researcher still owns source gathering and policy enforcement. The pipeline uses the shared global loading screen.
- Coco is a separate `gpt-5.6-luna` Responses API agent under `src/lib/orchestrator/agent/coco.ts`; its server route loads the Story's authoritative topic/synopsis and enables OpenAI's hosted `web_search` tool, returning safe source citations to the dialogue companion. The actor agent under `src/lib/orchestrator/agent/actor.ts` uses `ACTOR.md` to give map NPCs historically grounded, in-character greetings and replies.
- Generation runs are created at pipeline start and incrementally stored in `StoryGenRun`
  with free-text `steering`, the cleaned Researcher `topic`, Curator output, replay
  metadata, per-agent outputs, usage, and progress logs. Home and Steer persist each
  completed field as it becomes available; finalization only creates the playable Story
  and connects it to the existing run. Runs are listed under `/voyages` and displayed
  through the steer console at `/steer/:slug`; legacy rows without the new snapshots
  are hidden.
- Character portraits are reused by matching normalized character name plus `Character.ageRange`; shared portrait assets may be attached to multiple stories. The generated portrait `Asset.ageRange` records the same variant. A different age range always generates a separate portrait. Each Character may also have an optional `spriteAsset` linked to a reusable `CHARACTER_SPRITE` Asset; the star character uses this relation for its top-down sprite, independently of portrait age. A sprite asset keeps its default frame in `Asset.data` and may have any number of generic-keyed `AssetFrame` children for rotations or animations. Walk-through NPC sprite scale is derived from the sprite (or portrait) `Asset.ageRange` via `npcSpriteScale`.
- The old map prototype and its sprite pack are archived under `bin/src/app/`, `bin/src/lib/map/`, `bin/src/lib/game/`, and `bin/public/Cute_Fantasy/`; active map tile notes live in `.brain/MAP.md`.

## Target direction

Use a hybrid web architecture:

```text
Next.js/React UI
  ├─ dialogue, journal, sources, inventory, progression
  └─ Phaser 2D world
       ├─ movement, scenes, NPCs, interactions
       └─ map and simulation presentation

Shared scenario data → deterministic simulation engine → UI/world renderers
```

- Keep scenarios data-driven and independent of React or Phaser.
- Start text-first, then add visual-novel scenes and 2D exploration.
- Keep simulation rules deterministic; AI, if added, is a bounded presentation/helper layer.
- Store source, rights, and provenance metadata with content and assets.
- Use a PWA/web-first delivery model; consider native/3D only if the experience requires it.

The map walkthrough renderer is shared by `/draw/view` and `/home`. `/home` is a thin route wrapper around `src/app/components/game/GameController.tsx`, which owns map loading, input, camera, collision, NPC movement, and player-facing item interactions; `src/lib/game/npc/useNpcDialogue.ts` owns the NPC conversation state and actor calls, while `src/lib/game/interactions/` and `src/lib/game/items/useItemDialogue.ts` keep item interaction rules and dialogue data-driven. It loads the published `myroom` map and star-character summaries from the voyage API; star NPCs use `PLAYER_SPRITE_SCALE` and the same one-tile collision footprint as the player. NPC and item conversations are rendered by `MapRenderer` with the shared `DialoguePanel`; all NPC prompts, including free-form questions, call the actor boundary with a stable `Character.id`. `MapRenderer` also owns walkthrough fullscreen: `F` toggles browser fullscreen on the map root, with a bottom-right icon control. Story and other Coco surfaces use the generic `/api/coco/chat` boundary with prepared topic and dialogue context.
